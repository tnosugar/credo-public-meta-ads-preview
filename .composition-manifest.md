## Deviations from canonical (project tuning)
1. **ANCHOR_TAGS tightened** to block tags only (dropped inline `span/small/strong/em` and table/list tags) because the Facebook-feed markup keeps all text in `div`/`span` chrome; without this, reaction counts and the "Sponsored" label would light up. Meaningful anchors come from `[data-comment-target]`.
2. **localStorage fallback adapter** when `FIREBASE_CONFIG` is unset — lets the widget run for solo testing before the operator pastes Firebase credentials; banner shows a "Local-only" badge. Same 4-op interface as the RTDB adapter (per `firebase-rtdb-adapter.md` §"Adapter contract").
3. **Apply button removed** (reviewer-facing). The `pending -> applied` transition is NOT a reviewer action here — it happens externally: Claude harvests the comments from RTDB, runs a socratic apply pass on the page for each change request, then patches `status: 'applied'`. Pending rows show Edit / Archive / Delete; the Applied tab fills from the external pass. This is close to the library's `commentLifecycleMode: 'feedback-only'` but keeps the reviewer-facing **Archive** action, so it is a project variant rather than the stock feedback-only button set.
4. **Variant-aware anchors (per-dimension scoping).** The prototype carries A/B variants for three zones — headline (5 shared options), message (5 per ad), creative (Illustration / Photo / Still life / Video). The stock widget anchors a comment to a *slot*, so a comment would surface on every variant of that slot. To scope comments to the variant on screen, the prototype writes a semantic, selection-tracking `data-comment-id` and the widget re-matches on change:
   - headline → `hl-{idx}` (**campaign-wide**: same anchor across all ads showing that headline; the highlight pass uses `querySelectorAll` so every instance lights up).
   - message → `{ad}-msg-{idx}` (per ad).
   - creative → `{ad}-cr-{illustration|photo|stilllife|video}` (**direction only**; ratio is carried in `data-variant-note`, e.g. "Photo · 9:16", and surfaced as the comment's `text_preview` rather than splitting the anchor).
   - card → `{ad}-card`, description → `{ad}-desc` (no variant; unscoped).
   Widget edits backing this (3, small): (a) highlight pass `querySelector` → `querySelectorAll` for campaign-wide headlines; (b) `openComposer` prefers `data-variant-note` for `text_preview` so creative comments record direction+ratio instead of the literal `[creative]`; (c) `window.__rwRefresh = render` exposed in `init`, which the prototype calls after rewriting a zone's `data-comment-id` so the highlight pass re-runs against the now-current variant. Anchors no longer follow the canonical `{pageSlug}-{tag}-{n}` shape for these zones — they are pre-set and selection-driven. Harvest candidate below.

## Harvest candidate (surface to a workbench [spec-session]; do not promote from here)
The library's `comment-lifecycle.md` notes feedback-only mode where "an external process (Claude / a build script / a human operator) reads RTDB directly and patches status." It does NOT capture the specific Cowork workflow this project relies on: **Claude harvests widget comments, runs a two-way socratic questioning pass to apply each change request back onto the page, then moves the comment to the Applied column.** Worth atomizing upstream as the canonical "Claude-applied" lifecycle variant (a named `commentLifecycleMode`, e.g. `claude-applied`, that drops Apply but keeps Archive).

Second candidate — **variant-aware anchoring** (deviation 4). When a page renders multiple variants into one slot (A/B headline/message/creative behind dropdowns or tabs), comments should scope to the variant on screen, not the slot. The pattern here — selection-tracking `data-comment-id` + a `__rwRefresh` hook + `data-variant-note` for sub-dimension metadata that isn't part of the anchor key + `querySelectorAll` highlight for cross-instance (campaign-wide) anchors — generalizes. Worth atomizing as a `variant-anchors` strategy alongside `anchor-strategy.md`, with the per-dimension-vs-full-combination choice and the campaign-wide-vs-per-instance choice as documented forks.

Record these in `.claude/variations/review-widget.md` if they should persist as canonical-for-credo.
